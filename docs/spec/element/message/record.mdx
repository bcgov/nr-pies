Represents a message envelope for transmitting asset records.

## Scope

This message type serves as a transport envelope for transmitting records between systems. A Record message combines
Header metadata (transaction tracking, asset identification) with a resource payload selected via `record_kind`.

The Record message can be composed of either an [Anchor](/docs/spec/element/resource/anchor) or
[Permit](/docs/spec/element/resource/permit) resource type defined in the specification. Refer to the
[Type Hierarchy](/docs/spec#hierarchy) for how messages, resources and data types compose together.

## Structure

A Record message consists of all flattened [Header](/docs/spec/element/data/header) properties, along with a resource
payload selected via `record_kind`:

| Name            | Cardinality | Type                                     | Description & Constraints                                                    |
| --------------- | ----------- | ---------------------------------------- | ---------------------------------------------------------------------------- |
| (header fields) | (varies)    | [Header](/docs/spec/element/data/header) | Standard Header properties, including the `record_kind` selector (see below) |

**Resource Type Selection via `record_kind`:**

The `record_kind` field acts as a type selector, identifying which resource type this message contains:

- **record_kind: `ANCHOR`** - Message payload conforms to [Anchor](/docs/spec/element/resource/anchor) resource schema
- **record_kind: `PERMIT`** - Message payload conforms to [Permit](/docs/spec/element/resource/permit) resource schema

This selection is enforced at the schema level via conditional `if/then` logic.

## Implementation Guidance

### Type Selection Pattern

The Record message uses a type selector pattern:

- The `record_kind` field declares which resource type the payload contains
- Systems generating records must populate this field with the correct resource type name
- The values must match the actual resource schema of the message payload
- Schema validation uses `if/then` logic to enforce the match

### Resource-Specific Constraints

Each resource type that can be composed with Record defines its own constraints. Refer to the appropriate resource
documentation:

- [Anchor](/docs/spec/element/resource/anchor) - Foundational reference point
- [Permit](/docs/spec/element/resource/permit) - A permit record

Systems should provide all known element definitions whenever available. Recipient systems must not assume a single
message provides the full transactional state. Recipient systems should request additional Records if needed.

### Choosing a Resource Type

When generating a Record message, use `record_kind` to select the appropriate resource type:

Use record_kind `ANCHOR` when the message is a reference point without state:

- Establishing a baseline or foundational record
- Registering a record for future association
- Signaling record existence without process or hold context

Use record_kind `PERMIT` when the message carries process or hold state:

- Sharing a permit's journey through an application workflow
- Conveying hold periods and reasons for review delays
- Providing an auditable event history
