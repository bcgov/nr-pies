Represents a specific record.

## Scope

This message type is intended to represent an atomic Record originating from a source system. This will commonly be
used for sharing a record's process and on hold history. Use cases may include auditing, reporting or compliance.

Note that this message type does not guarantee nor expect all known attributes to be present. It is possible for this
message to only represent a subset of a canonically complete Record.

## Structure

A Record message consists of:

| Name            | Cardinality | Type                                     | Description                                                           |
| --------------- | ----------- | ---------------------------------------- | --------------------------------------------------------------------- |
| (header fields) | (varies)    | [Header](/docs/spec/element/data/header) | Standard Header properties, including Asset metadata (see note below) |

The `record_kind` field that determines which resource flavor applies should be part of the headers present in the message.

The `record_kind` value identifies which additional flavor-specific properties are present in the message payload. See the flavor documentation for details:

- **`Anchor`**: See [Anchor](/docs/spec/element/resource/anchor#structure)
- **`Permit`**: See [Permit](/docs/spec/element/resource/permit#structure)

## Implementation Guidance

### Message Transport and Kind Selection Pattern

The Record message uses a kind selection pattern to route between different resource types:

- The Asset within the [Header](/docs/spec/element/data/header) declares the resource type via `asset.record_kind`
- This kind selector determines which resource-specific schema applies (Permit, Anchor, etc.)
- The kind selector value must match the actual resource type in the message payload

### Kind-Specific Constraints

Different Record kinds define their own implementation requirements. Refer to the appropriate kind documentation:

- [Anchor](/docs/spec/element/resource/anchor) - A foundational reference point record
- [Permit](/docs/spec/element/resource/permit) - Tracks application process and events for permits

Systems generating this message should:

- Populate `header.asset.record_kind` with the correct resource type
- Ensure the kind selector value matches the actual resource type in the payload
- Provide all known element definitions associated with that resource kind whenever possible

Recipient systems must not assume that a single message provides the full data horizon. If a comprehensive dataset is required, request additional associated Records.
